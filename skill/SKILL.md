# Knowledge Exporter: 对话内容分析与归档

分析当前对话内容，自动识别所属研发环节，归类保存为结构化知识文档。

## Quick Reference

| 用户说 | 执行动作 |
|--------|---------|
| "保存对话" / "归档" | 分析当前对话 → 归类 → 生成文档 |
| "保存这段 bug 修复过程" | 分析指定内容 → 归档到 Bug 修复阶段 |
| "检查文档完整性" | 对照清单扫描已有文档，列出缺口 |
| "检查脚本独立性" | 扫描项目脚本 → 识别 AI 依赖 → 输出风险清单 |

---

## 核心流程

### Step 1 · 分析对话内容

回顾当前对话的完整内容，提取：

1. **做了什么**：具体完成的工作（写了脚本、修了 bug、做了设计...）
2. **关键知识**：对话中产生的有保存价值的信息（方案决策、分析过程、配置方法、踩坑经验...）
3. **涉及文件**：对话中创建或修改的文件列表

### Step 2 · 映射到研发阶段

将提取的内容对照 [references/asset-checklist.md](references/asset-checklist.md) 进行分类，确定属于哪个或哪几个研发阶段：

```
对话内容 → 属于哪个阶段？
  ├─ 需求理解、拆解、评估     → ① 需求分析
  ├─ 架构设计、方案决策、选型   → ② 设计
  ├─ 编码、编译、环境搭建、部署 → ③ 编码
  ├─ 单测、冒烟、自验          → ④ 自验
  ├─ Bug 定位、修复、性能优化   → ⑤ Bug 修复
  └─ 测试用例、自动化、测试环境 → ⑥ 测试
```

一次对话可能涉及多个阶段（如"修 bug 后补了单测"→ ⑤ + ④），全部识别出来。

### Step 3 · 生成文档并保存

对每个识别出的阶段，将对话中的相关内容整理为结构化文档：

**文档组织原则：**
- 同一阶段的内容保存到同一文档中（如多个 bug 追加到同一份 bug 记录）
- 如果目标文档已存在，**追加**新内容而非覆盖
- 文档结构由 AI 根据内容特点合理安排，不要生搬模板
- 默认保存到 `doc/` 目录，按阶段命名

**默认文件映射参考：**

| 阶段 | 默认文件路径 |
|------|------------|
| ① 需求分析 | doc/需求分析记录.md |
| ② 设计 | doc/设计决策记录.md |
| ③ 编码 | doc/编译构建指南.md 或按具体内容命名 |
| ④ 自验 | doc/自验测试记录.md |
| ⑤ Bug 修复 | doc/bug分析修复记录.md |
| ⑥ 测试 | doc/测试方法与记录.md |

CRITICAL: 文件映射仅供参考。根据对话实际内容灵活决定文件名和组织方式，同一阶段如果内容差异大，可拆分为多个文档。

### Step 4 · 向用户确认

保存后向用户展示：

```
📦 已归档当前对话内容：

| 研发阶段 | 保存位置 | 内容摘要 |
|---------|---------|---------|
| ⑤ Bug 修复 | doc/bug分析修复记录.md | 信号槽连接类型导致的属性数据丢失 |
| ③ 编码 | doc/编译构建指南.md | 新增 Release 打包流程 |

如需调整，请告诉我。
```

## 辅助功能 A：文档完整性检查

用户说"检查文档完整性"时，扫描 `doc/` 目录，对照 [references/asset-checklist.md](references/asset-checklist.md) 的 6 个阶段 31 项资产，列出已有和缺失的文档。

## 辅助功能 B：脚本独立性检查

用户说"检查脚本独立性"或"脚本能否独立运行"时，检查项目中的脚本在失去 AI 辅助后能否独立使用。

### 检查流程

#### B-Step 1 · 扫描项目脚本

扫描项目中所有脚本文件：
```
扫描范围：**/*.py, **/*.sh, **/*.bat, **/*.ps1, **/*.cmd, **/*.js, **/*.ts
排除目录：node_modules/, .venv/, venv/, __pycache__/, .git/
```

#### B-Step 2 · 识别 AI 依赖风险

对每个脚本进行三类检查：

**检查 1 — 代码中是否调用了 AI API：**

搜索以下关键词/模式：
```
openai, anthropic, claude, chatgpt, copilot, cursor,
ChatCompletion, chat.completions, llm, langchain,
transformers.pipeline, huggingface,
OPENAI_API_KEY, ANTHROPIC_API_KEY, AI_API
```

**检查 2 — 是否缺少使用文档：**

对每个脚本检查是否存在对应的使用说明（同目录的 README、脚本内的 `--help` 支持、`doc/` 中的说明文档）。

**检查 3 — 是否存在"未固化脚本"：**

分析当前对话历史，识别是否有"用户反复让 AI 临时生成但未保存为文件的脚本操作"。

#### B-Step 3 · 输出风险清单

```
🔍 脚本独立性检查结果：

| 脚本 | 风险类型 | 风险说明 | 建议操作 |
|------|---------|---------|---------|
| scripts/analyze.py | 🔴 AI API 调用 | 第42行调用 openai.chat.completions | 改为非 AI 方案或提供离线替代 |
| scripts/build.sh | 🟡 缺少文档 | 无使用说明，参数含义不明 | 补充参数说明和运行示例 |
| （对话中生成过但未保存） | 🟡 未固化 | 对话中多次让 AI 生成部署命令 | 固化为脚本文件并补充文档 |
| scripts/test.py | 🟢 无风险 | 无 AI 依赖，有 --help | 无需操作 |

总结：X 个脚本无风险，Y 个需要补充文档，Z 个存在 AI 依赖需要改造。
```

#### B-Step 4 · 协助修复

对于用户选择修复的项：
- **AI API 调用**：分析调用目的，提议用规则引擎、模板、或固定逻辑替代
- **缺少文档**：根据脚本代码直接生成使用说明文档
- **未固化脚本**：从对话历史中提取，保存为文件并补充文档

---

## 文档质量标准

- 内容来自对话中的实际工作，不编造
- 保留关键的分析过程和决策理由，不只记结论
- 操作步骤可直接执行
- 不含指向 Cursor 对话的引用
