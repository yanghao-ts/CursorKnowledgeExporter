# KnowledgeExporter 使用指南

将 Cursor 开发过程中的对话内容自动分析、归类，保存为结构化知识文档。

---

## 工具包内容

```
KnowledgeExporter/
├── install-skill.bat                     ← 一键安装脚本（Windows）
├── install-skill.sh                      ← 一键安装脚本（macOS/Linux）
├── 使用指南.md                           ← 本文件
├── cursor-knowledge-export-checklist.md  ← 人工参考版清单（可打印/离线用）
├── skill/
│   ├── SKILL.md                          ← Skill：对话内容分析与归档
│   └── references/
│       └── asset-checklist.md            ← 知识资产分类索引（6 阶段 31 项）
└── rule/
    └── knowledge-recording.mdc           ← Rule：完成工作后即时提议存档
```

---

## 快速开始（3 步）

### 第 1 步 · 安装到 Cursor

**安装 Skill**（全局生效，所有项目可用）：

方式一：运行安装脚本（推荐）
```
Windows: 双击 install-skill.bat
macOS/Linux: bash install-skill.sh
```

方式二：手动复制
```
将 skill/ 整个文件夹复制到：
  Windows: %USERPROFILE%\.cursor\skills\knowledge-export\
  macOS:   ~/.cursor/skills/knowledge-export/

最终目录结构：
  .cursor/skills/knowledge-export/
  ├── SKILL.md
  └── references/
      └── asset-checklist.md
```

**安装 Rule**（需要在每个项目中安装）：

```
将 rule/knowledge-recording.mdc 复制到目标项目的：
  <项目根目录>/.cursor/rules/knowledge-recording.mdc
```

### 第 2 步 · 主动存档（Skill）

在 Cursor 对话中说以下任意一句：

| 你说 | AI 做什么 |
|------|----------|
| "保存对话" / "归档" | 分析当前对话内容 → 识别研发阶段 → 生成文档并保存 |
| "保存这段 bug 修复过程" | 分析指定内容 → 归档到对应阶段 |
| "检查文档完整性" | 对照清单扫描已有文档，列出缺口 |

AI 会自动执行：
1. 分析当前对话内容，提取关键知识
2. 识别属于哪个研发阶段（编码/Bug修复/设计...）
3. 整理为结构化文档，保存到 `doc/` 目录
4. 向你确认保存结果

### 第 3 步 · 即时存档（Rule）

安装 Rule 后无需任何操作。当你完成一项具体工作后：

- 写完编译脚本并验证通过 → AI 提议："是否把编译步骤整理成指导文档？"
- 修复 bug 并验证通过 → AI 提议："是否把分析定位过程保存下来？"
- 确定了技术方案 → AI 提议："是否把方案对比分析记录下来？"

回复"好"/"保存"/"记录"，AI 就直接根据刚才的对话内容生成文档并保存。

## 使用场景

### 场景 A · 随时存档（推荐的日常用法）

1. 正常使用 Cursor 工作
2. 完成一项工作后，Rule 自动提议保存 → 回复"好"即可
3. 或者主动说"保存对话"让 Skill 执行完整归档

### 场景 B · 批量补档（截止日期前）

1. 打开一个有历史对话的 Cursor 项目
2. 说："保存对话"
3. AI 分析整个对话内容，按研发阶段归类生成文档
4. 在每个有价值的对话中重复此操作

> 小技巧：不同的 Cursor 对话有不同的上下文，建议在每个有价值的对话中都执行一次。

### 场景 C · 查漏补缺

1. 说："检查文档完整性"
2. AI 扫描 `doc/` 目录，对照 31 项资产清单
3. 列出哪些阶段已有文档、哪些还缺失
4. 选择需要补充的项，AI 会引导生成

## 31 项知识资产清单

AI 分析对话内容后，会将其映射到以下 6 个研发阶段的 31 项知识资产中：

### ① 需求分析（3 项）

| # | 资产名称 | 说明 |
|---|---------|------|
| 1.1 | 需求理解与拆解记录 | AI 帮你分析需求后的拆解结果、子任务清单 |
| 1.2 | 可行性分析 | 技术可行性评估、风险点识别、备选方案 |
| 1.3 | 工作量评估依据 | 让 AI 协助估算工作量时的评估逻辑 |

### ② 设计（4 项）

| # | 资产名称 | 说明 |
|---|---------|------|
| 2.1 | 架构/方案设计文档 | 模块划分、类关系、数据流、通信协议 |
| 2.2 | 关键设计决策及理由 | 为什么选 A 方案不选 B？trade-off 是什么？ |
| 2.3 | 接口/协议定义 | API 接口、消息格式、通信协议 |
| 2.4 | 技术选型对比 | 框架/库/工具的对比分析 |

### ③ 编码（7 项）

| # | 资产名称 | 说明 |
|---|---------|------|
| 3.1 | 编译/构建方法 | 从零编译的完整步骤 |
| 3.2 | 开发环境搭建指南 | IDE 配置、工具链、环境变量 |
| 3.3 | 依赖与第三方库清单 | 所有依赖库的名称、版本、获取方式 |
| 3.4 | 多平台构建差异 | 不同平台的构建差异和注意事项 |
| 3.5 | 部署/发布/打包流程 | 打包、签名、发布的完整步骤 |
| 3.6 | 代码关键逻辑注释 | 复杂逻辑的注释是否充分 |
| 3.7 | 代码架构约定 | 命名规范、模块职责、设计模式约定 |

### ④ 开发自验（4 项）

| # | 资产名称 | 说明 |
|---|---------|------|
| 4.1 | 自验测试方法 | 自测步骤、检查点、判定标准 |
| 4.2 | 单元测试用例 | 单测代码及覆盖范围 |
| 4.3 | 自验环境配置 | Mock 数据、测试桩、特殊配置 |
| 4.4 | 冒烟测试清单 | 提交前必须验证通过的核心场景 |

### ⑤ Bug 修复（5 项）

| # | 资产名称 | 说明 |
|---|---------|------|
| 5.1 | 历史 Bug 分析记录 | 现象→分析→根因→修复→验证 |
| 5.2 | Bug 分析方法论 | 定位问题的技巧和套路 |
| 5.3 | 高频 Bug 模式总结 | 反复出现的问题类型及预防措施 |
| 5.4 | 性能问题分析记录 | 瓶颈分析、基线数据、优化评估 |
| 5.5 | 调试方法与工具用法 | 调试技巧、日志分析方法、诊断命令 |

### ⑥ 测试（8 项）

| # | 资产名称 | 说明 |
|---|---------|------|
| 6.1 | 测试方法文档 | 各类测试的执行步骤和判定标准 |
| 6.2 | 测试用例定义 | 用例 ID、参数、预期结果、通过标准 |
| 6.3 | 测试环境搭建 | 环境配置、网络拓扑、设备连接 |
| 6.4 | 测试数据与基线 | 性能基线、容差范围、历史结果 |
| 6.5 | 测试结果判读方法 | 如何从日志判断 PASS/FAIL |
| 6.6 | 自动化脚本清单与说明 | 脚本位置、用途、参数、运行方式 |
| 6.7 | 自动化测试框架设计 | 框架架构、扩展方式、添加用例步骤 |
| 6.8 | 跨机器/跨平台协作机制 | 多端协作流程和脚本 |

> 完整清单含参考提示词见 `skill/references/asset-checklist.md`，也可查看 `cursor-knowledge-export-checklist.md` 获取可打印版本。

## 文档保存在哪里？

所有文档默认保存到项目的 `doc/` 目录，按研发阶段组织：

```
<项目根目录>/
└── doc/
    ├── 编译构建指南.md        ← ③ 编码阶段
    ├── bug分析修复记录.md     ← ⑤ Bug 修复阶段
    ├── 设计决策记录.md        ← ② 设计阶段
    ├── 测试方法与记录.md      ← ⑥ 测试阶段
    └── ...
```

文件名和组织方式由 AI 根据实际内容决定，上面只是参考。同一阶段的内容会追加到同一文档中，不会每次新建。

## 如何扩展知识资产类型

当你发现有内容不在现有 31 项清单中时：

### 方式一 · 让 AI 自动补充

```
我发现还需要记录"XX内容"，请补充到知识资产清单中
```

### 方式二 · 手动编辑

打开 `skill/references/asset-checklist.md`，在对应阶段的表格末尾追加一行。

## 团队推广建议

### 分发方式

1. 将 `KnowledgeExporter/` 放到团队共享目录或 Git 仓库
2. 每个成员运行 `install-skill.bat`（或 `.sh`）安装 Skill
3. 每个成员将 `knowledge-recording.mdc` 复制到自己的项目中

### 推广顺序

| 阶段 | 动作 |
|------|------|
| 第 1 天 | 分发工具包，每人安装 Skill + Rule |
| 第 1 周 | 日常工作中通过 Rule 提议即时存档 |
| 每周末 | 在重要对话中说"保存对话"做一次完整归档 |
| 截止前 | 说"检查文档完整性"做最终查漏补缺 |

### 质量检查

- [ ] 新人仅凭文档能独立完成对应工作
- [ ] 操作步骤可直接执行，不含模糊描述
- [ ] Bug 记录有完整链路：现象 → 分析 → 根因 → 修复 → 验证
- [ ] 文档内容不依赖 Cursor 对话

## 常见问题

**Q: Skill 和 Rule 有什么区别？**
- Skill：你**主动说**"保存对话"时触发，分析整个对话内容并归档
- Rule：工作**完成后自动**提议保存，你回复"好"就生成文档

**Q: 保存的文档会覆盖已有文档吗？**
- 不会。如果同类文档已存在，新内容会追加到已有文档中

**Q: 一次对话涉及多个研发阶段怎么办？**
- AI 会自动识别所有涉及的阶段，分别归档到对应文档

**Q: AI 生成的文档不准确怎么办？**
- 导出后务必人工审查，AI 生成的是初稿，细节需要确认和修正

**Q: 安装 Rule 后提议太频繁怎么办？**
- Rule 只在工作完成并验证通过后才提议，且可以直接忽略
- 如确实不需要，删除项目中的 `.cursor/rules/knowledge-recording.mdc`
