# Cursor 知识资产导出清单

> **背景**：2026-04-15 后团队将从 Cursor Team 切换至 GLM 模型。需在此之前将 Cursor 中积累的工作知识以文档形式导出保存，确保关键经验不丢失。
>
> **使用方法**：
> 1. 按研发流程的 6 个阶段逐项过，先在「是否用了 Cursor」列打勾
> 2. 打勾的阶段，逐条检查「需导出的知识资产」，标注状态
> 3. 对于需要导出的项，可直接复制「参考提示词」发给 Cursor 执行导出
> 4. 状态标记：✅ 已导出 / ⏳ 进行中 / ❌ 未开始 / ➖ 不涉及
>
> **适用角色**：开发人员、测试人员（各角色关注的阶段不同，见下表）

## 导出执行指南

### 提示词使用技巧

| 技巧 | 说明 |
|------|------|
| **在原项目对话中执行** | 提示词应在包含对应工作历史的 Cursor 对话/项目中发送，AI 才能回顾上下文 |
| **分批导出** | 不要一次发所有提示词，按阶段逐批执行，确认质量后再继续下一批 |
| **补充上下文** | 如果 AI 回顾不到相关历史，补充说明："我们在 X 月做过 XX 功能，当时讨论了 YY 问题" |
| **指定保存路径** | 提示词中已包含 `保存到 doc/xxx.md`，可根据项目实际目录结构调整 |
| **质量审查** | 导出后检查文档内容是否准确、完整，有遗漏的地方追问 AI 补充 |

### 优先级建议

| 优先级 | 阶段 | 理由 |
|--------|------|------|
| P0 立即 | ⑤ Bug 修复 + ③ 编码（编译构建） | 这些知识最难重新获取，直接影响日常工作 |
| P1 本周 | ② 设计 + ⑥ 测试 | 设计决策和测试方法是核心资产 |
| P2 下周 | ① 需求分析 + ④ 自验 | 相对容易重建，但有则更好 |

### 质量自检

每份导出的文档，自问：
- [ ] 离开 Cursor 后，仅凭此文档，新人能独立完成对应工作吗？
- [ ] 文档中的操作步骤，是否可直接执行（而非"大概是这样"）？
- [ ] Bug 记录是否有完整链路：现象 → 分析 → 根因 → 修复 → 验证？
- [ ] 文档中没有指向 Cursor 对话的引用（这些将不可访问）？
